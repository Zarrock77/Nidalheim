# Architecture

Vue d'ensemble du système Nidalheim et des flux entre composants.

## Monorepo

```
nidalheim/
├── services/
│   ├── api-auth/      # Express 5 REST (auth)
│   ├── api-game/      # ws WebSocket (IA temps-réel)
│   ├── db/            # Drizzle schéma + migrations
│   ├── site/          # Next.js vitrine
│   └── docs/          # Nextra docs admin
└── infra/
    ├── docker-compose.yml
    ├── nginx/         # reverse proxy
    ├── p4d/           # Perforce (assets UE5) — stack Docker séparé
    └── .env.example
```

Pas de workspace racine — chaque package gère ses deps indépendamment (mixed `pnpm`/`npm`).

## Services

| Service | Runtime | Port | Responsabilité |
|---------|---------|------|---------------|
| `api-auth` | Node 22 / TS | 3001 | Register, login, refresh, logout — JWT HS256 |
| `api-game` | Node 22 / JS ESM | 3002 | WebSocket `/text` et `/audio` — PNJ IA |
| `db-migrate` | one-shot | — | Applique les migrations Drizzle |
| `nidalheim-site` | Next.js 16 | 3000 | Site vitrine + register/login UI |
| `nidalheim-docs` | Next.js 16 | 3000 | Documentation admin-only |
| `postgres` | Postgres 16 | 5432 | Users, player profiles, chat history |
| `redis` | Redis 7 | 6379 | Refresh tokens (TTL 7j avec rotation) |
| `nginx` | nginx alpine | 80, 443 | Reverse proxy + TLS direct pour `api-game` |
| `certbot` | certbot/certbot | — | Renouvelle le cert Let's Encrypt en boucle (12h) |

## Reverse proxy (Nginx)

Un seul conteneur Nginx route vers les domaines prod et staging :

```
www.nidalheim.com       → nidalheim-site:3000           (Cloudflare → :80)
docs.nidalheim.com      → nidalheim-docs:3000           (Cloudflare → :80)
api-auth.nidalheim.com  → nidalheim-api-auth:3001       (Cloudflare → :80)
api-game.nidalheim.com  → nidalheim-api-game:3002       (TLS direct → :443)
www-staging.nidalheim.com       → host.docker.internal:3013   (Cloudflare → :80)
docs-staging.nidalheim.com      → host.docker.internal:3014   (Cloudflare → :80)
api-auth-staging.nidalheim.com  → host.docker.internal:3011   (Cloudflare → :80)
api-game-staging.nidalheim.com  → host.docker.internal:3012   (TLS direct → :443)
```

Le bloc `api-game` active `proxy_http_version 1.1` + `Upgrade/Connection` pour le WebSocket, et `proxy_read_timeout 86400s` pour tenir les connexions voice de longue durée.

### Stratégie TLS hybride

| Domaine | Cloudflare | Terminaison TLS |
|---------|------------|-----------------|
| `www`, `docs`, `api-auth`, root | Proxied (orange) | Cloudflare termine, tape origin en HTTP/80 |
| `www-staging`, `docs-staging`, `api-auth-staging` | Proxied (orange) | Cloudflare termine, tape origin en HTTP/80 |
| `api-game`, `api-game-staging` | DNS-only (gris) | **Nginx du VPS termine via Let's Encrypt** |

`api-game` bypass volontairement Cloudflare pour éviter ses limites WebSocket en plan free (idle timeout ~100s, buffering qui ajoute de la latence sur le streaming audio Cartesia/Deepgram). Le cert est obtenu via challenge HTTP-01 webroot et renouvelé automatiquement par le service `certbot` défini dans `docker-compose.yml` (boucle `certbot renew` toutes les 12h).

Voir [Déploiement / Reverse proxy Nginx](/docs/backend/deploy#reverse-proxy-nginx) pour le bootstrap initial du cert.

## Données partagées

### PostgreSQL

Une seule base `nidalheim`, accédée par :
- `api-auth` → tables `users`, `player_profiles`
- `api-game` → table `chat_messages` (à créer, cf [Services / Database](/docs/backend/services/database))
- `db-migrate` → gère le schéma

### Redis

Utilisé uniquement par `api-auth` pour le stockage des **refresh tokens** avec TTL de 7 jours. Keys format : `refresh:<userId>:<tokenId>`.

## Séparation auth

Deux niveaux d'auth indépendants :

1. **`api-auth` ↔ UE5 / `services/site`** : JWT HS256, signé par `api-auth`, vérifié par `api-game`
2. **`services/docs` ↔ utilisateur** : NextAuth credentials provider qui fait lui-même un `POST /login` sur `api-auth`, puis gate `role === "admin"`

La clé `JWT_SECRET` est partagée entre `api-auth` et `api-game` via `infra/.env`. `services/docs` n'en a pas besoin (il parle à `api-auth` en HTTP directement).

## Décisions architecturales

| Décision | Pourquoi |
|----------|----------|
| JWT **HS256** (vs RS256) | 2 services contrôlés en interne → la symétrique est suffisante et simplifie l'opérationnel (pas de PEM à gérer) |
| Monorepo **non-workspace** | Chaque service a sa propre lifecycle ; pas de besoin de sharing de code cross-service |
| **Drizzle** (vs Prisma) | Plus léger, génération SQL explicite, type-safe sans codegen runtime |
| **Groq** (vs OpenAI) pour le LLM | Llama 70B à 10x la vitesse d'OpenAI, coût < 0.1$/M tokens |
| **Cartesia** (vs ElevenLabs) TTS par défaut | Latency first-chunk < 400ms, connexion WS persistante → pas de handshake par tour |
| **Registry-based deploy** (vs build-on-VPS) | Build parallèle sur GH runners, rollback par SHA, pas de charge CPU pendant le deploy |
| Convention **Lyra-style** côté UE5 | Reconnaissable par tout dev UE5, scalable par domaine fonctionnel — sans adopter les patterns lourds (GameFeatures, ExperienceDefinitions). Détails [Client UE5 → Conventions](/docs/unreal-engine/conventions) |
| **Multi-instance NPC** (UE5) | Chaque PNJ posé en niveau a son propre `NpcId` + identité backend (system_prompt, voice, conversation history scopée `(user_id, npc_id)`). Remplace le pattern singleton initial — détails [NPCCharacter](/docs/unreal-engine/characters/npc) |

## Client UE5

Le client jeu vit dans un **depot Perforce séparé** (`main`, hébergé sur le VPS Epitech via `infra/p4d/`). Il consomme `api-auth` (Device Authorization Grant) et `api-game` (WebSocket text + audio). Voir la section [Client UE5](/docs/unreal-engine/overview) pour le source layout, les conventions Lyra adoptées, le système PNJ multi-instance et le flow d'auth côté C++.
