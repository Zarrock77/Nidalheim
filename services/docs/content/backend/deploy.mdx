# Deploiement

Le pipeline CI/CD est defini dans `.github/workflows/deploy.yml`.

Il a deux chemins separes :

- push sur `main` : build/push Docker vers GHCR, puis deploiement production via Docker Compose.
- push sur `staging` : update du checkout `~/Nidalheim-staging`, installation des deps, migrations DB staging. Aucun restart automatique des scripts staging.

La staging ne build pas d'images Docker. Elle tourne en process host avec les scripts `start-staging.sh` pour permettre une iteration rapide sur les domaines `*-staging.nidalheim.com`.

## Branches

| Branche | Runtime | Checkout VPS | Base DB | Domaine |
|---------|---------|--------------|---------|---------|
| `main` | Docker Compose | `~/Nidalheim` | `${POSTGRES_DB}` | `*.nidalheim.com` |
| `staging` | scripts dev lances manuellement | `~/Nidalheim-staging` | `${POSTGRES_DB}_staging` | `*-staging.nidalheim.com` |

La prod et la staging partagent le meme VPS, le meme Postgres Docker et le meme Redis Docker, mais pas la meme base Postgres. La staging utilise Redis DB 1 dans `api-auth/start-staging.sh`.

## Production (`main`)

### Build

Le job `build` s'execute seulement sur `main`.

Il build et push 5 images :

| Service | Context Docker | Image |
|---------|----------------|-------|
| site | `./services/site` | `ghcr.io/zarrock77/nidalheim-site` |
| docs | `./services/docs` | `ghcr.io/zarrock77/nidalheim-docs` |
| api-auth | `./services/api-auth` | `ghcr.io/zarrock77/nidalheim-api-auth` |
| api-game | `./services/api-game` | `ghcr.io/zarrock77/nidalheim-api-game` |
| db-migrate | `./services/db` | `ghcr.io/zarrock77/nidalheim-db-migrate` |

Chaque image est taguee deux fois :

- `:latest` : derniere prod publiee par `main`.
- `:<full-sha>` : version immuable utilisee par le deploy et le rollback.

Il n'y a pas de tag `:prod-latest` : dans ce setup, `:latest` est deja le tag prod. Un tag `:staging-latest` n'aurait du sens que si la staging consommait aussi des images Docker.

### Deploy

Le job `deploy-production` :

1. se connecte au VPS en SSH ;
2. met `~/Nidalheim` a jour sur `origin/main` en fast-forward ;
3. ecrit `~/Nidalheim/infra/.env` depuis les GitHub Secrets, avec `IMAGE_TAG=${{ github.sha }}` ;
4. lance `docker compose pull` ;
5. lance `docker compose up -d --remove-orphans` ;
6. restart nginx ;
7. execute `docker compose run --rm db-migrate` ;
8. prune les images Docker inutilisees.

Le compose utilise `image: ghcr.io/zarrock77/nidalheim-<service>:${IMAGE_TAG:-latest}`. En deploy CI, `IMAGE_TAG` pointe donc sur le SHA exact qui vient d'etre build.

## Staging (`staging`)

Le job `deploy-staging` ne depend pas du job Docker `build`.

Il :

1. se connecte au VPS en SSH ;
2. clone `~/Nidalheim-staging` si le checkout n'existe pas encore ;
3. realigne ce checkout sur `origin/staging` si le worktree est propre ;
4. ecrit `~/Nidalheim-staging/infra/.env` depuis les GitHub Secrets ;
5. cree la base `${POSTGRES_DB}_staging` si besoin ;
6. execute les migrations Drizzle sur cette base ;
7. installe les dependances des services staging ;
8. ne restart aucun process staging.

Scripts staging :

| Script | Port host | Domaine |
|--------|-----------|---------|
| `services/api-auth/start-staging.sh` | `3011` | `api-auth-staging.nidalheim.com` |
| `services/api-game/start-staging.sh` | `3012` | `api-game-staging.nidalheim.com` |
| `services/site/start-staging.sh` | `3013` | `www-staging.nidalheim.com` |
| `services/docs/start-staging.sh` | `3014` | `docs-staging.nidalheim.com` |

Les scripts staging calculent leur racine avec `NIDALHEIM_ROOT` ou, par defaut, avec leur emplacement sur disque. Pour isoler la prod, ils doivent etre lances depuis `~/Nidalheim-staging`, pas depuis `~/Nidalheim`.

## Domaines et TLS

Production :

| Domaine | Upstream | TLS |
|---------|----------|-----|
| `www.nidalheim.com` | `nidalheim-site:3000` | Cloudflare proxied |
| `docs.nidalheim.com` | `nidalheim-docs:3000` | Cloudflare proxied |
| `api-auth.nidalheim.com` | `nidalheim-api-auth:3001` | Cloudflare proxied |
| `api-game.nidalheim.com` | `nidalheim-api-game:3002` | VPS direct Let's Encrypt |

Staging :

| Domaine | Upstream | TLS |
|---------|----------|-----|
| `www-staging.nidalheim.com` | `host.docker.internal:3013` | Cloudflare proxied |
| `docs-staging.nidalheim.com` | `host.docker.internal:3014` | Cloudflare proxied |
| `api-auth-staging.nidalheim.com` | `host.docker.internal:3011` | Cloudflare proxied |
| `api-game-staging.nidalheim.com` | `host.docker.internal:3012` | VPS direct Let's Encrypt |

`api-game` et `api-game-staging` doivent rester en DNS-only sur Cloudflare. Le VPS termine TLS directement pour eviter les limitations WebSocket/audio de Cloudflare free.

## Bootstrap VPS

Production, une seule fois :

```bash
ssh vps
cd ~
git clone git@github.com:Zarrock77/Nidalheim.git Nidalheim
cd ~/Nidalheim/infra
chmod +x init-letsencrypt.sh
./init-letsencrypt.sh
```

Staging, une seule fois apres que `nginx.conf` contient les blocs staging :

```bash
ssh vps
cd ~/Nidalheim/infra
chmod +x init-letsencrypt-staging.sh
./init-letsencrypt-staging.sh
```

Les scripts staging se lancent ensuite a la main depuis `~/Nidalheim-staging/services/<service>/start-staging.sh`. Les units systemd restent disponibles dans `infra/systemd/` comme option future, mais le workflow ne les restart pas.

## Migrations DB

Production :

```bash
cd ~/Nidalheim/infra
docker compose run --rm db-migrate
```

Staging :

```bash
cd ~/Nidalheim-staging/services/db
DATABASE_URL="postgresql://<user>:<password>@localhost:5432/<db>_staging" pnpm migrate
```

Le workflow staging cree la base avant migration :

```bash
docker exec nidalheim-postgres sh -c "createdb -U '<user>' '<db>_staging' 2>/dev/null || true"
```

## Rollback production

Sur le VPS :

```bash
cd ~/Nidalheim/infra
IMAGE_TAG=<sha-anterieur> docker compose up -d
```

Le rollback staging se fait en repointant la branche `staging` sur le commit voulu puis en pushant `staging`. Le workflow refuse d'ecraser un worktree staging avec des modifications locales non committees.

## Commandes utiles

Logs staging si les scripts sont lances manuellement dans un terminal ou un tmux :

```bash
cd ~/Nidalheim-staging/services/api-game && ./start-staging.sh
cd ~/Nidalheim-staging/services/api-auth && ./start-staging.sh
cd ~/Nidalheim-staging/services/site && ./start-staging.sh
cd ~/Nidalheim-staging/services/docs && ./start-staging.sh
```

Option systemd staging :

```bash
cd ~/Nidalheim-staging/infra
./setup-staging-systemd.sh
systemctl --user restart nidalheim-api-game-staging.service
```

Relancer le workflow sans push :

```bash
gh workflow run deploy.yml --ref main
gh workflow run deploy.yml --ref staging
```

## Limites et ameliorations

- Pas encore de job `check` avant deploy : ajouter lint/typecheck/build pour bloquer les deploys cassants.
- Pas encore de healthcheck post-deploy : ajouter des `curl` sur les 8 domaines prod/staging.
- La staging ne restart pas les scripts automatiquement : les watchers doivent etre lances depuis `~/Nidalheim-staging`.
- La staging partage le meme Postgres physique que la prod : la base est separee, mais il faudra des backups et une vigilance sur les migrations destructives.
