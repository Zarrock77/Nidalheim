## Protocole client вҶ” server

### Client вҶ’ Server (WebSocket)

| Type | Payload | Description |
|------|---------|-------------|
| JSON control | `{ type: "audio_config", sample_rate: 48000, device: "...", bluetooth_hfp: false }` | **PremiГЁre frame obligatoire**. Permet Г  Deepgram d'ГӘtre ouvert au bon sample rate. |
| PCM base64 | `"<base64>"` (>100 bytes) | Chunk audio mono, 16-bit LE |
| Control | `"COMMIT"` | Force la finalisation de l'utterance courante (push-to-talk release) |

### Server вҶ’ Client (JSON uniquement)

| `type` | `data`/`message` | Description |
|--------|------------------|-------------|
| `user_transcript_partial` | `"bonj..."` | Transcript interim Deepgram (affichable en typing) |

| `text` | `"<full NPC reply>"` | RГ©ponse complГЁte du LLM (aprГЁs `onComplete`) |

| `error` | `"stt: ...", "llm: ...", "tts: ..."` | Erreur d'un composant |

## DГ©roulement d'un tour

1. **Connexion WS** avec `?token=<JWT>` вҶ’ auth HS256
2. **`audio_config`** reГ§u вҶ’ ouverture Deepgram au bon sample rate, dГ©marrage TTS Cartesia en parallГЁle (warm-up)
3. **Hydration** (parallГЁle, non-bloquant) : `ConversationStore.loadRecent(userId, 20)` вҶ’ injectГ© dans le contexte LLM
4. **Stream PCM** вҶ’ Deepgram Г©met `onInterim` (partial) et `onFinal` (speech_final)
5. **`COMMIT`** reГ§u (ou `speech_final` de Deepgram) вҶ’ accumulation du transcript
6. **`_flushUtterance`** : appelle `llm.streamResponse(userText)`
   - Chaque delta token вҶ’ `tts.sendText(delta)` вҶ’ Cartesia Г©met du PCM en streaming
   - `onComplete` вҶ’ `tts.flush()` + persist du tour (user + assistant) en transaction Postgres
7. L'utilisateur peut reparler вҶ’ retour Г©tape 4

## Latency budget

Sur une utterance courte ("salut") :

| Step | Latency observГ©e |
|------|------------------|
| STT final (Deepgram) | 100-200 ms aprГЁs fin d'Г©locution |
| LLM first token (Groq Llama 70B) | 150-300 ms |
| TTS first chunk (Cartesia persistent) | 200-400 ms |

## Hot path вҖ” optimisations

### 1. Pas d'ouverture Deepgram avant `audio_config`

Le client envoie son `sample_rate` dans la premiГЁre frame JSON. On attend cette info avant d'ouvrir Deepgram pour Г©viter un reconfigure au premier chunk audio (qui produit des transcripts vides sur les premiers 200ms).

Pendant l'attente, les chunks PCM reГ§us sont mis en buffer (capped Г  ~1s d'audio @ 48kHz pour Г©viter l'OOM).

### 2. Cartesia persistent WS

Ouvert une fois dans `pipeline.start()`. Chaque tour :
```js

tts.sendText(delta);       // streaming
tts.flush();               // finalise la synthГЁse
```

### 3. Hydrate + TTS warmup en parallГЁle

```js

  const history = await conversationStore.loadRecent(user.id);
  if (history.length > 0) llm.setHistory(history);

  await ttsRoute.instance.start();  // warm Cartesia
}

```

Aucun des deux ne bloque le premier PTT.

### 4. Audio config mid-session

Si le user switche de micro (Bluetooth HFP вҶ” micro intГ©grГ©), le client renvoie un nouveau `audio_config` avec le nouveau sample rate. `AudioPipeline.onClientAudioConfig` dГ©tecte le changement et fait un `stt.reconfigure()` transparent вҖ” pas besoin de relancer l'Г©diteur UE5.

## Gestion d'erreurs

- **STT error** вҶ’ `{ type: "error", message: "speech-to-text: ..." }`

**Exception** : Г©chec du `pipeline.start()` вҶ’ `clientWs.close()` avec un message d'erreur JSON avant fermeture.
