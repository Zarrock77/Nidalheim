# api-auth

Service REST d'authentification. Express 5 + TypeScript + npm.
Écoute sur le port **3001**.

## Stack

- **Express 5** (routing, middleware)
- **jsonwebtoken** — signature et vérif HS256
- **bcrypt** — 12 rounds pour les mots de passe
- **ioredis** — refresh tokens store (TTL 7 jours)
- **pg** — driver PostgreSQL natif (pas de Drizzle côté runtime, SQL direct)
- **cors** — origines configurables via `CORS_ORIGINS`

## Endpoints

### `POST /register`

Crée un nouveau compte.

**Requête** :
```json
{
  "username": "Zarrock",
  "email": "user@example.com",
  "password": "min-8-chars"
}
```

**Réponses** :
- `201 Created` → `{ accessToken, refreshToken }`
- `400 Bad Request` → champs manquants ou password < 8 chars
- `409 Conflict` → username ou email déjà utilisé

Crée aussi une ligne dans `player_profiles` avec les valeurs par défaut (xp=0, level=1).

### `POST /login`

Identifiant accepté : **username OU email** (le champ s'appelle `username` mais le lookup fait `WHERE username = $1 OR email = $1`).

**Requête** :
```json
{
  "username": "Zarrock",     // ou "user@example.com"
  "password": "..."
}
```

**Réponses** :
- `200 OK` → `{ user: { id, username, email, role }, accessToken, refreshToken }`
- `401 Unauthorized` → credentials invalides
- `400 Bad Request` → champs manquants

### `POST /refresh`

Rotation du refresh token : l'ancien est révoqué dans Redis, un nouveau couple est émis.

**Requête** : `{ refreshToken }`

**Réponses** :
- `200 OK` → `{ accessToken, refreshToken }`
- `401 Unauthorized` → token invalide, expiré ou révoqué

### `POST /logout`

Révoque le refresh token côté Redis. Idempotent : même avec un token invalide, renvoie `200`.

**Requête** : `{ refreshToken }`

**Réponse** : `200 OK` → `{ message: "Logged out successfully" }`

### `GET /health`

Liveness probe. Toujours `200` tant que le process tourne.

**Réponse** : `{ status: "ok", service: "nidalheim-auth" }`

## Device Authorization Grant (RFC 8628)

Pour le client UE5 qui ne peut pas afficher de formulaire de login confortablement, on expose un flow OAuth standard où le jeu ouvre un browser vers `www.nidalheim.com/device` pour que le joueur approuve.

Voir **[Authentification](/docs/auth#device-flow)** pour le diagramme complet et les détails.

### `POST /device/authorize` *(no auth)*

Émis par UE5 au démarrage de l'écran de connexion.

**Body** : `{}`

**Réponse 200** :
```json
{
  "device_code": "<64 hex chars opaque>",
  "user_code": "ABCD-EFGH",
  "verification_uri": "https://www.nidalheim.com/device",
  "verification_uri_complete": "https://www.nidalheim.com/device?code=ABCD-EFGH",
  "expires_in": 600,
  "interval": 5,
  "interval_ms": 1000
}
```

`user_code` utilise un alphabet sans caractères ambigus (pas de `I`, `O`, `0`, `1`).

### `POST /device/token` *(no auth)*

Polled par UE5 toutes les `interval_ms` ms.

**Body** : `{ "device_code": "..." }`

**Réponses possibles** :
- `200 OK` → `{ user, accessToken, refreshToken }` (one-shot, le record est consommé)
- `400` + `{ "error": "authorization_pending" }` → poll continuer
- `400` + `{ "error": "slow_down" }` → augmenter l'interval
- `400` + `{ "error": "access_denied" }` → l'user a refusé
- `400` + `{ "error": "expired_token" }` → 10 min écoulées, le record n'existe plus

### `POST /device/lookup` *(Bearer auth)*

Appelé par la page `/device` une fois le user loggé, pour vérifier que le code existe avant d'afficher la confirmation.

**Body** : `{ "user_code": "ABCD-EFGH" }`

**Réponse 200** : `{ user_code, client_name, expires_in }`

### `POST /device/approve` *(Bearer auth)*

Appelé quand le user clique le bouton **Approve** sur la page `/device`.

**Body** : `{ "user_code": "ABCD-EFGH" }`

**Réponse 200** : `{ "status": "approved" }`

Mint un **nouveau couple de tokens** pour la session device (séparés du couple utilisé pour le web), les stocke dans le record Redis. Le poll `/device/token` suivant les renvoie au client UE5.

### `POST /device/deny` *(Bearer auth)*

Symétrique d'`approve`. Marque le record comme `denied`. Le poll `/device/token` suivant retournera `access_denied`.

### Stockage Redis

| Key | Valeur | TTL |
|-----|--------|-----|
| `device:<device_code>` | JSON `{ status, user_code, expires_at, [user, access_token, refresh_token] }` | 600s |
| `usercode:<user_code>` | `<device_code>` (réverse lookup pour la page web) | 600s |

Le record est **consommé** (DELETE Redis) au premier `/device/token` qui retourne 200 — pas de rejeu possible.

## JWT

- Algorithme : **HS256**
- Secret : `process.env.JWT_SECRET` (partagé avec `api-game`)
- Access token expiry : `JWT_ACCESS_TOKEN_EXPIRY` (default `15m`)
- Refresh token expiry : `JWT_REFRESH_TOKEN_EXPIRY` (default `7d`)

### Payload access token

```ts
{
  sub: "uuid-user",   // user id
  username: "Zarrock",
  role: "player" | "admin",
  iat: 1714000000,
  exp: 1714000900     // +15m
}
```

### Payload refresh token

```ts
{
  sub: "uuid-user",
  tokenId: "uuid-random",
  iat: 1714000000,
  exp: 1714604800     // +7d
}
```

Le `tokenId` est stocké en Redis sous `refresh:<userId>:<tokenId>` avec TTL 7j. Refresh = vérif du JWT + check Redis + delete + insert nouveau couple.

## Flow d'auth côté UE5

```
1. POST /register → { accessToken (15m), refreshToken (7d) }
2. Stocker les deux tokens (NidalheimAuthStorage)
3. Utiliser accessToken pour wss://api-game/?token=<jwt>
4. Si accessToken expire → POST /refresh → nouveau couple
5. Déco : POST /logout pour invalider le refresh
```

## CORS

La liste des origines autorisées est lue depuis `CORS_ORIGINS` (CSV) :

```bash
CORS_ORIGINS=https://www.nidalheim.com,https://docs.nidalheim.com,http://localhost:3000
```

Default si absente : `https://www.nidalheim.com,http://localhost:3000`.

## Sécurité

- **bcrypt 12 rounds** sur les mots de passe
- **JWT signé HS256** avec un secret 64 bytes aléatoire
- **Refresh tokens rotatifs** (réutiliser un refresh token révoqué → 401)
- **Idempotent logout** (pas de leak d'info sur la validité d'un token)
- **Express 5 + cors + express.json** standards
