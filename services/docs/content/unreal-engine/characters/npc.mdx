# NPCCharacter

Système des PNJ pilotés par IA. Deux classes complémentaires :

- **`ANidalheimNPCCharacter`** — le "cerveau" : double WebSocket text+audio, identité backend, audio capture/playback
- **`ANidalheimInGameNPC`** — l'instance posée en niveau : InteractionSphere qui déclenche le proximity chat

## Hiérarchie

```
ACharacter (Unreal)
└── ANidalheimCharacterBase           (GAS shared)
    └── ANidalheimNPCCharacter        (cerveau, double WebSocket vers api-game)
        └── ANidalheimInGameNPC       (instance posée en niveau, InteractionSphere)
```

## Multi-instance — refactor récent

> ⚠️ Le pattern singleton `ANidalheimNPCCharacter::GetInstance()` a été supprimé. Chaque PNJ posé en niveau a sa propre identité backend.

### Identité par PNJ

```cpp
// Source/Nidalheim/Character/NidalheimNPCCharacter.h
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "NPC")
FString NpcId = TEXT("default");
```

Le `NpcId` matche la table `npcs` côté `api-game` (cf [Database](/docs/backend/services/database)). Chaque entrée a son propre :
- `system_prompt` — prompt LLM personnalisé
- `voice_id` — voix Cartesia ou ElevenLabs
- `llm_model` — modèle LLM par défaut

L'historique conversationnel est scopé `(user_id, npc_id)` côté Postgres → switcher de PNJ ne mélange pas les conversations.

### Construction de l'URL WebSocket

```cpp
wss://api-game.nidalheim.com/text?token=<JWT>&npc=<NpcId>
wss://api-game.nidalheim.com/audio?token=<JWT>&npc=<NpcId>
```

`api-game` vérifie le JWT en HS256 avec le `JWT_SECRET` partagé (cf [Auth backend](/docs/backend/auth#jwt-hs256)). Si `npc` est absent ou inconnu, fallback sur le seed `default` inclus dans la migration `0002_*`.

```cpp
// BuildWsUrl
FString ANidalheimNPCCharacter::BuildWsUrl(
    const FString& BaseUrl,
    const FString& AccessToken,
    const FString& NpcId)
{
    if (BaseUrl.IsEmpty() || AccessToken.IsEmpty()) return BaseUrl;
    FString Url = BaseUrl;
    {
        const TCHAR Joiner = Url.Contains(TEXT("?")) ? TEXT('&') : TEXT('?');
        Url = FString::Printf(TEXT("%s%ctoken=%s"), *Url, Joiner,
                              *FGenericPlatformHttp::UrlEncode(AccessToken));
    }
    if (!NpcId.IsEmpty())
    {
        Url = FString::Printf(TEXT("%s&npc=%s"), *Url,
                              *FGenericPlatformHttp::UrlEncode(NpcId));
    }
    return Url;
}
```

### Switcher de PNJ runtime

```cpp
UFUNCTION(BlueprintCallable, Category = "NPC")
void SetNpcId(const FString& NewNpcId);
```

Closes les deux WebSockets et reopens avec le nouveau `?npc=<id>`. No-op si `NewNpcId == NpcId`.

## Pipeline audio

Détails dans [Voice pipeline](/docs/backend/voice-pipeline) côté backend. Côté UE5 :

1. **Capture** : la capture push-to-talk passe par **miniaudio** (header vendored `ThirdParty/miniaudio/`, WASAPI), qui ouvre un device au sample rate système avec une latence bien moindre que l'ancien `UAudioCaptureComponent`
2. **Streaming** : envoi binaire frames PCM au WebSocket `/audio` (le client envoie un `audio_config` au début pour signaler le sample rate)
3. **Commit** : `CommitAudioBuffer()` envoie un message texte `COMMIT` qui finalise l'utterance côté Deepgram
4. **Playback** : les frames PCM TTS reçues (24 kHz mono) sont écrites dans un ring buffer miniaudio (`ma_pcm_rb`) lu par un device de playback miniaudio — la lecture ne passe pas par l'AudioMixer UE

La capture ET le playback sont donc entièrement gérés par **miniaudio** (latence 150–400 ms évitée vs `UAudioCaptureComponent` + AudioMixer/USynthComponent).

> ⚠️ **Refactor récent** : `UNidalheimAudioCaptureComponent` (et le module `AudioCapture`) ont été **supprimés** — la capture est désormais gérée par **miniaudio** (latence 150–400 ms évitée vs AudioMixer + USynthComponent).

## Proximity chat (`ANidalheimInGameNPC`)

Le pattern type pour un PNJ posé en niveau :

```
ANidalheimInGameNPC
  ├── InteractionSphere (USphereComponent, rayon 500cm par défaut)
  └── NpcId = "olaf"  (configuré en BP)

Joueur entre dans la sphère
  → ChatSubsystem->ShowChatWithNpc(NpcId)
  → NPCCharacter->SetNpcId(NpcId) si différent
  → ChatWidget visible

Joueur sort
  → ChatSubsystem->HideChat()
```

Le NPCCharacter sous-jacent reste connecté ; seule la visibilité de l'UI et l'identité backend changent.

### Création d'un nouveau PNJ

1. **Côté backend** : INSERT dans la table `npcs` côté `services/db` (migration ou seed) avec `id`, `system_prompt`, `voice_id`, `llm_model`
2. **Côté UE5** : créer un BP qui hérite de `ANidalheimInGameNPC`, set `NpcId` au même identifiant, le poser dans le niveau
3. Optionnel : ajuster `InteractionRadius` selon la taille de la zone d'interaction souhaitée

## Lifecycle

```
BeginPlay
  → Init capture audio miniaudio (device WASAPI)
  → ConnectTextWebSocket / ConnectAudioWebSocket
       └─ AuthSubsystem->EnsureFreshAccessToken
              └─ OpenTextSocketWithToken / OpenAudioSocketWithToken
                     └─ BuildWsUrl + FWebSocketsModule::Get().CreateWebSocket

EndPlay
  → bShuttingDown = true
  → StopAudioCapture
  → Close WebSockets
```

## Events serveur & synchronisation missions

Sur le canal texte, le PNJ distingue une réplique (texte brut) d'un **event JSON** (message commençant par `{`) — un event n'est jamais affiché dans le chat. Events gérés (`TryHandleServerEvent`, aussi traités sur le canal audio) :

| Event | Effet client |
|---|---|
| `mission_started` | `QuestSubsystem->MarkMissionStarted` → HUD objectifs visible, état persisté |
| `mission_validation_result` | `MarkMissionCompleted` + consommation de l'objet + demande d'expansion (`dungeon_expand`) |
| `dungeon_extension` | fusion/validation du plan étendu (`ApplyExtensionInWorld`) → le donjon s'étend en place |
| `user_transcript` / `user_transcript_partial` | transcription affichée dans le chat |
| `audio` | chunk PCM TTS → ring buffer de playback |

Dans l'autre sens, le PNJ pousse au backend : `mission_sync` (état des missions, à la connexion et à chaque `OnMissionsChanged`), `inventory_sync` (inventaire entier, à chaque changement), `dungeon_expand` (après une validation), `clear_history` (reset). Détails du flux : [Donjon & missions](/docs/unreal-engine/quests).

## Reconnect logic

- **Canal texte** : reconnexion **toutes les 500 ms** tant que la socket n'est pas rétablie (`ScheduleTextReconnect`). Les messages **techniques** (syncs, `dungeon_expand`, `clear_history`) émis pendant une coupure sont mis en file et partent à la reconnexion (`FlushPendingTextSends`). Les **messages de chat du joueur ne sont jamais mis en file** : envoyés hors connexion, ils sont perdus (le joueur réessaie) — un texte rejoué en différé produirait une réponse hors contexte.
- **Canal audio** : une reconnexion est retentée une fois (`bAudioReconnectAttempted`) après une fermeture serveur.
- À chaque reconnexion texte : `mission_sync` + `inventory_sync` complets, et reprise d'une extension de donjon manquante (`TryResumeDungeonExpand`).
