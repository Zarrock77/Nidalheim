# Client UE5 — Aperçu

Le client jeu Unreal Engine 5.6 (`Nidalheim.uproject`) vit dans un **depot Perforce séparé** du monorepo backend (cf [Workflow Perforce](/docs/unreal-engine/perforce)). Il consomme les services backend documentés dans le reste de cette doc :

- **`api-auth`** via REST + Device Authorization Grant pour l'auth joueur ([détails](/docs/unreal-engine/auth))
- **`api-game`** via WebSocket `/text` et `/audio` pour les PNJ pilotés par IA ([voice pipeline](/docs/backend/voice-pipeline)) et la **génération de quêtes contextuelles** ([Quêtes](/docs/unreal-engine/quests))

Le module de gameplay est entièrement en **C++** (convention `Nidalheim*`), les Blueprints ne portant que la config visuelle, l'input, les animations et les callbacks.

> ⚠️ **Divergence avec l'archi EIP documentée.** Le document de contexte Cowork décrit un `AC_AIBrain` (Actor Component faisant des appels REST IA + TTS) et un GAS en Blueprint. L'**implémentation réelle** diverge : le GAS est **100 % C++**, et toute l'IA (LLM + STT + TTS + génération de quête) passe par `api-game` via WebSocket — pas d'appel REST direct UE5 → LLM/TTS, pas d'`AC_AIBrain`. Cette doc décrit le code réel.

## Build.cs — modules

**Public** : `Core`, `CoreUObject`, `Engine`, `InputCore`, `EnhancedInput`, `AIModule`, `StateTreeModule`, `GameplayStateTreeModule`, `GameplayTasks`, `GameplayTags`, `GameplayAbilities`, `UMG`, `Slate`, `Sockets`, `WebSockets`, `AudioMixer`, `Niagara`, `NiagaraCore`, `ChaosNiagara`, `NetCore`, `NavigationSystem`, `AssetRegistry`, `AnimGraphRuntime`, `CinematicCamera`.

**Privé** : `SSL`, `HTTP`, `SlateCore`, `Json`, `JsonUtilities`.

**Éditeur seulement** (`if (Target.bBuildEditor)`) : `UnrealEd`, `AnimGraph`, `BlueprintGraph` — requis par les [commandlets de maintenance d'assets](/docs/unreal-engine/commandlets), jamais linkés en build game/client.

> ⚠️ Le module `AudioCapture` a été **retiré** : la capture push-to-talk passe désormais par **miniaudio** (header vendored dans `ThirdParty/miniaudio/`, WASAPI uniquement), qui remplace `UAudioCaptureComponent` (latence 150–400 ms via le buffering AudioMixer + USynthComponent).

`PublicIncludePaths` inclut les sous-dossiers pour permettre les `#include "X.h"` sans préfixe de chemin.

> ⚠️ **Include paths morts** : `Nidalheim/Audio`, `Nidalheim/Quest` et `Nidalheim/Quest/Objectives` sont encore listés dans `Build.cs` mais ces dossiers **n'existent pas** (le quest subsystem vit dans `System/`). `Nidalheim/AbilitySystem/Attributes` y figure aussi en double. À nettoyer.

## Configuration runtime

`Config/DefaultGame.ini`, section `[Nidalheim]` :

```ini
AuthApiUrl="https://api-auth.nidalheim.com"
WebVerificationUrl="https://www.nidalheim.com/link"
WebsocketTextBaseUrl="wss://api-game.nidalheim.com/text"
WebsocketAudioBaseUrl="wss://api-game.nidalheim.com/audio"
AudioCaptureVolume=0.1
```

Pour pointer vers un backend local (cf [Setup](/docs/backend/setup-project)) : remplacer par `http://localhost:3001` / `ws://localhost:3002/text` etc.

## Coordination cross-repo backend ↔ UE5

Toute modification de protocole doit être appliquée des **deux côtés** :

| Modification | Côté backend | Côté UE5 |
|---|---|---|
| Forme du JWT (claims) | `services/api-auth/src/services/jwt.ts` + `api-game/src/auth.js` | `FNidalheimAuthSubsystem` (lecture des claims) |
| Query params WebSocket | `services/api-game/src/server.js` | `BuildWsUrl` dans `NidalheimNPCCharacter.cpp` |
| Endpoints device flow | `services/api-auth/src/routes/device.ts` | `FNidalheimAuthClient` |
| Schema `npcs` | `services/db/src/schema/npcs.ts` + migration | `NpcId` UPROPERTY (juste un FString, pas de breaking) |
| Format de quête (`quest_offer`) | `services/api-game` (`QuestGenerator`) | `UNidalheimQuestSubsystem` (réception JSON) |
| `JWT_SECRET` rotation | Mettre à jour `infra/.env` sur les deux services | Force re-login (tous les access tokens existants invalidés) |
